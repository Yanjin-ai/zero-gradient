# Zero-Gradient 项目：路线与设计

> 配套竞赛：[The Post-Backprop Challenge: Zero-Gradient Learning for Efficiency](https://www.kaggle.com/competitions/the-post-backprop-challenge-zero-gradient-learning-for-efficiency)
> 本文档是 `zero gradient.md`（思考笔记）的落地版：把约束、策略、架构-算法协同、路线图固定下来。
> 最后更新：2026-06-25

---

## 0. 一句话定位

在**单张 T4 16GB / 3 小时 / 零全局梯度**的约束下，自研一个**可局部训练的语言模型**（小而真的主干 + 稀疏专家库凑够 4B），用**手写的局部学习规则**训练它，做到 **perplexity 能看 + 显存/速度击败 BP**，并产出**可复现、可审计**的公开 Kaggle Notebook。

核心叙事：**BP 在 T4 上训 4B 直接 OOM，我们的局部方法能跑且 loss 下降。**

---

## 1. Kaggle 硬性要求（已核实）

来源：官方 `NOTE.md` + 公开高票/最新 kernel（2026-06-24 核实）。这是一个 **Hackathon，提交是公开 Notebook，不是自动榜**。

| 维度 | 硬性要求 |
|---|---|
| **参数量门禁** | 训练前模型必须**常驻 ≥ 4,000,000,000 fp16 参数**（≈8GB）。是被强制的，不是愿景。 |
| **零梯度** | 必须完全替换 BPTT 和**所有** `torch.autograd` / `loss.backward()` / 标准优化器（Adam/SGD），改用数学自洽的**手写局部规则**（Forward-Forward / Predictive Coding / Target Propagation / deeply-supervised local heads）。提交通常全局 `torch.set_grad_enabled(False)`。 |
| **局部梯度** | 局部目标的梯度可以**手算**（如 softmax-CE 的 `δ = p − onehot` 是标准允许技巧），只要不调 autograd、不跨块全局回传。 |
| **硬件/时间** | 单张 Tesla T4 16GB **或** 4 核 CPU；**3 小时**内完成整个训练循环。 |
| **可复现** | 强确定性：固定 `SEED=42`、`torch.use_deterministic_algorithms(True)`，两次运行结果一致。 |
| **数据** | **自备**（官方"无提供数据集"）。真实评测基准 = WikiText-103（perplexity）+ HellaSwag/PIQA（accuracy），有 offline gate：数据没挂上不得声称官方分数。 |
| **产物** | `/kaggle/working/` 下需有 loss 曲线、memory profile、`run_summary.json` 等可审计 artifact。 |

### 评判实质
不是看绝对 SOTA perplexity，而是看：**规则合规性 + 系统证据（峰值显存 / tokens/s / loss 曲线）+ 可复现 + 写作清晰可信**。最高票（21 票）那篇靠的是**诚实、清晰、可审计**的小模型，而非刷分。

## 2. 为什么有这些要求（用意 → 反推打法）

这些约束是**精心设计来卡死 BP、给局部方法留缝**的：

- **为什么禁全局梯度/BP**：BP 是大规模效率瓶颈——为 backward 须**缓存全部 activations**（吃显存）、backward **严格序列**（难并行）、forward+backward **算两遍**（慢）。挑战在问：局部学习能否绕开这三个枷锁。
- **为什么偏偏 4B + 单 T4（题眼）**：BP 训 4B 需 权重8GB + 梯度8GB + Adam动量16GB + activations ≫ T4 的 16GB，**直接 OOM 跑不起来**；而 4B 稀疏 fp16 权重仅 8GB，塞得进。**这个组合是故意选的"BP 必死、局部方法有活路"临界点。** → 我们最强的对比叙事。
- **为什么 3h / 弱机**：逼算法效率而非堆算力；平民化、人人可复现。
- **为什么真 LM / perplexity**：序列建模是 credit assignment 最难的真实任务，能在此 work 才说明局部方法有料。

**反推出的打法**：展示 `BP 训 4B 在 T4 OOM（截图为证）` → 展示 `我们的局部方法在同机跑通且 loss 下降`。绕开"必须刷出有竞争力绝对 perplexity"的陷阱。

## 3. 架构与算法必须协同设计

与 BP 最本质的区别：

- **BP 架构无关**：写任意可微前向，autograd 自动 backward → 可以先搭模型后管训练。
- **零梯度局部规则不是架构无关**：每块"局部目标是什么"、块怎么连（残差污染局部性）、有没有 attention（跨位置混合 → 位置间局部 credit assignment 更难）——都直接决定更新公式。

**结论：算法和架构必须一起设计。原创性正长在这个接缝上。**

### 复用 vs 自研
- **可复用（不违规）**：tokenizer（现成 BPE / tiktoken）、数据（公开语料）、NN 标准算子（attention/FFN/embedding 的数学）。
- **必须自研**：① 块划分的架构、② 手写的局部更新规则、③ 怎么合法撑到 4B。
- **绝不能用**：任何预训练权重（都是 BP 训出来的，违背立意且大概率违规）。

## 4. 目标架构：小而真的主干 + 大而稀疏的专家库

A（合规优先）与 B（质量优先）的合题，也直接落地笔记里"不同层不同预算"的思想：

```
[真·主干]   6–12 层小 Transformer（d≈512–768，~几亿参数）
            → 承载语言建模质量，出真实 WikiText-103 perplexity
            → 高预算：每块强局部规则（deeply-supervised local head）

[稀疏专家库] 大量真实专家，被"学习式路由"真正用到（不是哈希废库）
            → 承载 4B 参数门禁
            → 低预算：每 token 只稀疏激活几个，轻量局部更新
```

- **主干** = 质量来源（出 perplexity）。
- **专家库** = 满足 4B + 被真实使用 + 轻微改善。
- 天然就是 **Dynamic Layerwise**：主干高预算强规则，专家稀疏低预算，importance routing 决定本步更新哪些专家。

### 主干局部规则（已验证可行的存在性证明）
**Deeply-Supervised Nets**（每块挂 next-token head）：
- 局部目标：`L_ℓ = −log p_ℓ(y | h_ℓ)`，`p_ℓ = softmax(h_ℓ Wₗ_head + bₗ_head)`
- 闭式局部梯度（无 autograd）：`δ_ℓ = p_ℓ − onehot(y)`
- 手写更新：`ΔW = −η · h_{ℓ-1}ᵀ δ`（必要时穿过本块一层 relu'）
- 数学自洽：算的是**局部目标的精确梯度**（线性+softmax 手推平凡），只是不往全局传。

### 现实预期（要诚实）
**4B 在 3h/T4 内不会"收敛成好模型"——这是物理限制，没人做得到。** perplexity 来自小而真的主干；专家库主要是"被正确使用 + 轻微改善 + 满足门禁"。故事是"在 4B/零梯度/3h/T4 约束下跑通真实可学的 LM 并击败 BP 的显存/速度"，而非"训出强 4B"。

## 5. 与 `zero gradient.md` 笔记的冲突点

笔记整体框架（分层异构预算、top-k importance routing、稀疏更新、关键层强规则/非关键层弱更新、EMA 稳定、三档存储）**全部兼容且方向正确**。需要修正的只有重要性信号：

- **真冲突**：笔记里多个"层重要性"信号基于梯度——**梯度范数、Fisher 近似（=E[梯度²]）、SHAP grad×activation**。零梯度下根本算不出梯度，**全部弃用**。
- **替换为无梯度代理**：activation norm/variance、局部预测 mismatch（局部 loss）、FF 的 positive-negative goodness gap、局部 error 的 EMA。（笔记其实已预感到这点。）
- **半冲突**："偶尔全局扫描/校正"——全局 **backward** 违规；全局 **forward** 探测（跑前向收集各层统计）合法。重新定义为"偶尔全局**前向**探测"即可。

**一句话**：架构级思想没冲突，只需把"梯度系"重要性度量换成"局部误差/激活系"。

## 6. 本机 vs Kaggle 分工

本机环境：torch 2.10 + MPS + tiktoken + transformers，10 核 / 17GB，**无 CUDA**。

- **本机（Phase 0）**：验证**逻辑、正确性、确定性、算法是否真能学、路由是否承重**。显存/速度/OOM 数字本机不算数。
- **Kaggle T4（Phase 1）**：同一份**配置驱动**代码，扩到 ≥4B、挂真 WikiText-103，跑真实显存/速度 + BP-OOM 演示 + 3h 守卫。
- **原则**：写一次，扩大只改配置数字，不重写。

## 7. 路线图

### Phase 0 — nano 纵切片（本机，五脏俱全）

最终提交需要的每个部件，这一版都先有 nano 版，先打通整条线。

| 最终部件 | nano 版 | 验证什么 |
|---|---|---|
| A. 确定性/复现框架 | seed + grad 全局关 + 跑两遍对比 | 两遍结果一致 |
| B. tokenizer + 数据管线 | 真 BPE + 小段真实文本 → (上文,下一token) | 接口与最终一致 |
| C. 块结构架构 + 4B 门禁 | 小主干(2块) + 迷你专家库(8专家) + 参数计数断言 | 可局部训练、门禁可用 |
| D. 手写前向(暴露每块状态) | nano 前向，无 autograd | 局部更新拿得到状态 |
| E. 手写局部更新规则 | deeply-supervised：`δ=p−onehot`,`ΔW=hᵀδ` | 零 autograd 下 loss 真降 |
| F. 无梯度重要性 + 稀疏路由 | activation/局部误差打分 + top-k + EMA | 路由可用且**有用** |
| G. 训练循环 | 时间守卫 + 显存追踪钩子 + monitor loss | 整条循环跑通 |
| H. 评测 | perplexity + next-token accuracy | 出有限、能看的数 |
| I. BP baseline + "4B OOM"演示 | 微型 BP 对照 harness | 对比框架就位（真 OOM 在 Kaggle） |
| J. 产物 | loss 曲线 / memory profile / run_summary.json | 可审计 |
| K. 合规自检 | 断言训练路径无 backward/autograd、≥门禁、确定性 | 一键证明合规 |

**搭建顺序（每步有过关门）**：
1. 骨架 + 确定性框架（A、K-确定性）。
2. 数据管线（B）：真 BPE + 小段 WikiText → 窗口 + train/val 切分。
3. nano 模型 + 参数计数门禁（C）：配置驱动，如 d=64、2 主干块、8 专家 top-2；门禁阈值本机设小，Kaggle 设 4e9。
4. 手写前向暴露状态（D）。
5. **纯 deeply-supervised 局部更新先跑通（E）**：先不路由，确认"零 autograd 下 monitor loss 严格降、val ppl 有限且 < unigram"。地基，必须先过。
6. **叠加无梯度重要性 + top-k 稀疏专家路由 + EMA（F）**：关键过关条件——**importance 路由的 val ppl 优于"随机路由"对照**。这是整个项目的研究信号：证明动态路由**承重**。
7. 训练循环（G）：时间守卫 + 显存追踪钩子。
8. 评测（H）。
9. 微型 BP 对照 harness（I）：小模型用 autograd 训，测显存/step 时间；写好"4B OOM 演示"函数。
10. 产物 + 合规断言（J、K）。

**Phase 0 总过关条件**（全在本机，全绿才扩大）：
- 两遍跑结果一致（确定性）。
- 训练路径零 autograd（`requires_grad=False` / 无 backward）。
- monitor loss 严格下降。
- val perplexity 有限且 < unigram baseline。
- **importance 路由 > 随机路由**（路由承重）。
- 产物齐全、门禁函数可用。

### Phase 1 — Kaggle T4 扩大（同代码改配置）
- 专家库配置撑到 ≥4B（如 4 banks × 4096 experts × 512²）。
- 挂真 WikiText-103，报真实 perplexity + 常识 MC accuracy。
- 跑真实峰值显存 / tokens·s⁻¹。
- 跑 **BP-OOM 演示**（同机训 4B BP → 截 OOM）。
- 加 3h wall-clock 守卫，确认确定性复现。

### Phase 2 — 差异化与写作
- 消融：去掉 importance routing / 固定 top-k / 不同 score 定义 / 不同存储档。
- 把"BP OOM vs 我们能跑"做成核心图。
- Notebook 写作：清晰、诚实、可审计（决定票数/评判）。

## 8. 待定 / 下一步决策

- **头牌卖点**：是"真实 perplexity 能看"（主干做大做好、专家库简化）还是"动态 importance routing 本身"（专家库+路由做主角）？决定工程力气往哪压。
- **专家库与路由如何具体咬合**（Phase 0 第 6 步）：路由信号、top-k、专家更新粒度的精确定义。
- attention 参数是否也局部更新（难）还是固定/随机（reservoir 式，简单）。
